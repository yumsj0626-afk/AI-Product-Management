## 한 줄 요약

ML 프로젝트는 일반 소프트웨어 프로젝트보다 불확실성이 크고, 데이터 품질·모델 성능·사용자 신뢰·운영 이후 성능 저하까지 관리해야 합니다. 그래서 ML 프로젝트는 단순 구현보다 **문제 정의 → 데이터 이해 → 데이터 준비 → 모델링 → 평가 → 배포/운영**의 반복적 프로세스로 관리해야 합니다.

---

# 1. ML Projects vs. Software Projects

## 핵심 개념

머신러닝은 최종적으로 사용자에게는 소프트웨어처럼 보입니다. 하지만 ML 프로젝트는 일반 소프트웨어 프로젝트와 관리 방식이 꽤 다릅니다.

일반 소프트웨어는 요구사항을 구현했는지 비교적 명확하게 판단할 수 있습니다. 반면 ML 프로젝트는 모델 성능이 확률적이고, 처음부터 어느 정도 성능이 나올지 알기 어렵습니다.

```text
일반 소프트웨어 프로젝트
→ 요구사항을 만족하는 기능을 구현하는 문제

ML 프로젝트
→ 데이터와 모델을 통해 충분히 좋은 성능을 만드는 문제
→ 성능이 100%가 될 수 없고, "얼마나 좋으면 충분한가?"를 정해야 함
```

---

## ML 프로젝트가 더 어려운 이유

| 구분 | 일반 소프트웨어 프로젝트 | ML 프로젝트 |
| :--- | :--- | :--- |
| 결과물 | 기능 구현 | 모델이 포함된 제품/서비스 |
| 성공 기준 | 요구사항 충족 여부가 비교적 명확 | 성능이 확률적이라 기준 설정이 필요 |
| 진행 방식 | 대체로 앞으로 진행 | 실험 결과에 따라 앞뒤로 오감 |
| 핵심 리스크 | 일정, 기능 구현, 버그 | 데이터 품질, 모델 성능, 환경 변화 |
| 팀 구성 | 개발자 중심 | 데이터 사이언티스트, ML 엔지니어, 도메인 전문가 등 필요 |
| 배포 후 관리 | 버그 수정, 기능 개선 | 모델 모니터링, 재학습, 성능 저하 관리 필요 |

---

## ML 프로젝트의 주요 관리 난점

### 1. 더 넓은 팀 역량이 필요하다

ML 프로젝트는 단순 개발자만으로 끝나기 어렵습니다.

```text
소프트웨어 엔지니어
데이터 사이언티스트
ML 엔지니어
도메인 전문가
제품/비즈니스 담당자
사용자 또는 고객 담당자
```

역할이 다양해질수록 협업 난이도도 높아집니다. 특히 데이터, 모델, 제품, 비즈니스 목표가 서로 연결되어 있기 때문에 팀 간 커뮤니케이션이 중요합니다.

---

### 2. 기술적 리스크가 높다

ML 프로젝트를 시작할 때는 보통 다음을 정확히 알 수 없습니다.

```text
어떤 데이터가 유용한지
어떤 feature가 중요한지
어떤 알고리즘이 적합한지
모델 성능이 어느 정도까지 나올지
데이터 품질 문제가 얼마나 클지
```

이 불확실성 때문에 일정, 예산, 성과를 예측하기 어렵습니다.

---

### 3. 프로젝트 진행이 직선적이지 않다

일반 소프트웨어 프로젝트는 보통 앞으로 나아가는 방향입니다. 속도가 빠르냐 느리냐의 차이는 있어도 대체로 기능이 쌓입니다.

하지만 ML 프로젝트는 다릅니다.

```text
모델 성능 개선
→ 다른 feature 추가
→ 오히려 성능 하락
→ 데이터 문제 발견
→ 다시 전처리 단계로 복귀
→ 다시 모델링
```

즉, 한 걸음 앞으로 갔다가 두 걸음 뒤로 갈 수도 있습니다. 이 때문에 관리자나 임원에게 진행 상황을 설명하기도 어렵습니다.

---

### 4. 모델은 확률적이다

ML 모델은 보통 100% 정확한 결과를 만들 수 없습니다. 그래서 중요한 질문은 이것입니다.

> “얼마나 좋은 성능이면 충분한가?”

이를 위해서는 적절한 error metric을 정하고, 비즈니스 가치를 만들 수 있는 최소 성능 기준을 설정해야 합니다.

---

### 5. 데이터 작업이 대부분의 시간을 차지한다

ML 프로젝트에서는 모델링보다 데이터 작업이 더 많은 시간을 차지하는 경우가 많습니다.

```text
데이터 소스 찾기
데이터 접근 권한 확보
데이터 수집
결측치 처리
이상치 확인
데이터 정제
feature 생성
feature 선택
데이터 품질 검증
```

강의에서는 흔히 ML 프로젝트 시간의 약 80%가 데이터 준비와 정제에 쓰이고, 실제 모델링과 튜닝에는 약 20% 정도가 쓰인다고 설명합니다.

---

### 6. 배포 후에도 끝나지 않는다

모델은 배포 후에도 계속 관리해야 합니다. 현실의 환경은 계속 변하기 때문입니다.

```text
사용자 행동 변화
시장 환경 변화
데이터 분포 변화
시스템 입력 변화
운영 조건 변화
```

이런 변화가 생기면 모델 성능이 떨어질 수 있습니다. 따라서 배포 이후에도 성능 모니터링과 재학습 계획이 필요합니다.

---

### 7. 사용자의 신뢰와 변화 관리가 필요하다

ML 제품은 단순히 새로운 도구를 추가하는 수준이 아니라, 사용자의 업무 흐름 자체를 바꿀 수 있습니다. 사용자가 모델을 신뢰하지 않으면 모델을 쓰지 않고, 모델을 쓰지 않으면 비즈니스 가치도 만들어지지 않습니다.

따라서 ML 프로젝트에서는 기술적 배포뿐 아니라 다음도 중요합니다.

```text
사용자 교육
업무 프로세스 변화 관리
모델 결과에 대한 설명
모델 신뢰 형성
```

---

# 2. Why Have a Data Science Process?

## 핵심 개념

엔지니어는 문제를 보면 바로 해결책을 만들고 싶어 합니다. 하지만 ML 프로젝트에서 문제 정의와 데이터 준비를 건너뛰면, 잘못된 문제를 풀거나 품질 낮은 데이터로 모델을 만들게 됩니다.

> Garbage in, garbage out.  
> 데이터가 쓰레기면 아무리 모델을 잘 만들어도 결과도 좋을 수 없습니다.

그래서 ML 프로젝트에는 체계적인 프로세스가 필요합니다.

---

## 프로세스가 필요한 이유

| 이유 | 설명 |
| :--- | :--- |
| 문제 정의 오류 방지 | 잘못된 문제를 풀면 시간과 비용이 낭비됨 |
| 작업 순서 관리 | 데이터 준비 전에 모델링부터 하면 품질 문제가 발생 |
| 팀 역할 정리 | 각 단계별 책임과 역할을 명확히 할 수 있음 |
| 반복 관리 | 실험과 수정이 많은 ML 프로젝트를 구조화할 수 있음 |
| 리스크 감소 | 데이터, 모델, 배포 리스크를 단계적으로 확인 가능 |

---

# 3. CRISP-DM Data Science Process

## CRISP-DM이란?

CRISP-DM은 **Cross Industry Standard Process for Data Mining**의 약자입니다. 데이터 마이닝과 머신러닝 프로젝트를 위한 대표적인 프로세스입니다.

강의에서는 CRISP-DM이 산업에 상관없이 적용 가능한 유연한 방법론이며, 오늘날에도 널리 쓰이는 데이터 과학 프로젝트 방법론이라고 설명합니다.

---

## CRISP-DM의 6단계

```text
1. Business Understanding
2. Data Understanding
3. Data Preparation
4. Modeling
5. Evaluation
6. Deployment
```

중요한 점은 CRISP-DM이 선형 프로세스가 아니라는 것입니다. 각 단계는 반복될 수 있고, 새롭게 배운 내용에 따라 이전 단계로 돌아갈 수 있습니다.

```text
문제 정의
→ 데이터 이해
→ 데이터 준비
→ 모델링
→ 평가
→ 배포
→ 사용자 피드백 / 환경 변화
→ 다시 문제 정의 또는 데이터 단계로 복귀
```

---

# 4. CRISP-DM 단계별 정리

## Step 1. Business Understanding

Business Understanding은 “무엇을 만들 것인가?”보다 먼저 “어떤 문제를 왜 풀 것인가?”를 정의하는 단계입니다.

### 이 단계에서 해야 할 일

```text
타깃 사용자가 누구인지 정의
사용자가 겪는 구체적 문제 정의
현재 사용자가 문제를 어떻게 해결하는지 파악
현재 방식의 한계와 gap 확인
성공의 의미 정의
비즈니스 impact를 metric으로 연결
제약 조건 확인
모델링에 중요한 요인 후보 식별
```

---

## 문제 정의에서 중요한 질문

| 질문 | 의미 |
| :--- | :--- |
| 누가 문제를 가지고 있는가? | 타깃 사용자 정의 |
| 그 사용자는 어떤 pain point를 겪는가? | 해결할 문제 정의 |
| 지금은 어떻게 해결하고 있는가? | 현재 상태 파악 |
| 현재 방식의 한계는 무엇인가? | ML이 들어갈 수 있는 gap 확인 |
| 성공하면 어떤 impact가 생기는가? | 비즈니스 가치 정의 |
| 어떤 제약 조건이 있는가? | 시간, 비용, 규제, 운영 조건 확인 |

---

## Outcome Metric vs Output Metric

Business Understanding 단계에서는 성공을 측정할 metric을 정해야 합니다. 이때 metric은 크게 outcome metric과 output metric으로 나눌 수 있습니다.

| 구분 | 의미 | 예시 |
| :--- | :--- | :--- |
| Outcome Metric | 사용자가 실제로 얻는 비즈니스 결과 | 복구 시간 감소, 비용 절감, 고객 만족도 향상 |
| Output Metric | 모델 자체의 기술적 성능 | MSE, accuracy, precision, recall |

중요한 것은 output metric이 좋아지는 것만으로는 충분하지 않다는 점입니다. 모델 성능이 실제 비즈니스 결과로 이어져야 합니다.

---

## Step 2. Data Understanding

Data Understanding은 문제 해결에 필요한 데이터를 수집하고, 그 데이터가 실제로 쓸 수 있는지 이해하는 단계입니다.

### 이 단계에서 해야 할 일

```text
데이터 소스 식별
데이터 수집
타깃 label 확보
feature 후보 생성
데이터 품질 검증
결측치, 오류, 이상치 확인
EDA 수행
feature와 target 간 관계 파악
feature 간 관계 파악
```

---

## 데이터 이해에서 자주 나오는 문제

| 문제 | 설명 |
| :--- | :--- |
| Missing Data | 필요한 값이 비어 있음 |
| Erroneous Data | 잘못 입력된 값이 있음 |
| Outliers | 일반적인 패턴에서 벗어난 값이 있음 |
| Label 부족 | 학습에 필요한 정답 데이터가 부족함 |
| 데이터 소스 불일치 | 여러 데이터가 서로 다른 기준으로 저장되어 있음 |

---

## Step 3. Data Preparation

Data Preparation은 모델링에 사용할 수 있도록 데이터를 정리하고 변환하는 단계입니다.

### 이 단계에서 해야 할 일

```text
train / test split
feature set 정의
feature 생성
feature 결합
feature selection
categorical variable encoding
scaling
class imbalance 처리
중복되거나 불필요한 feature 제거
```

데이터 준비 단계는 모델 성능에 큰 영향을 줍니다. 이 단계를 제대로 하지 않으면 이후 모델링에서 아무리 시간을 써도 좋은 결과를 얻기 어렵습니다.

---

## Step 4. Modeling

Modeling은 다양한 알고리즘을 실험하고, 모델을 훈련하고, 하이퍼파라미터를 조정하는 단계입니다.

### 이 단계에서 해야 할 일

```text
여러 알고리즘 비교
cross-validation 수행
모델 성능 비교
hyperparameter tuning
단순성과 복잡성의 균형 찾기
실험 기록
버전 관리
최종 모델 선택
```

ML 모델링은 정해진 공식대로 한 번에 끝나는 과정이 아닙니다. 여러 방법을 시도하고, 무엇이 잘 되고 무엇이 안 되는지 기록하면서 개선하는 탐색 과정입니다.

---

## 실험 기록과 버전 관리가 중요한 이유

ML 프로젝트에서는 여러 모델, 여러 feature set, 여러 하이퍼파라미터 조합을 실험합니다. 기록하지 않으면 무엇을 해봤고, 무엇이 효과 있었는지 잊어버리기 쉽습니다.

```text
어떤 데이터 버전을 썼는가?
어떤 feature를 사용했는가?
어떤 알고리즘을 사용했는가?
어떤 hyperparameter를 적용했는가?
성능은 어땠는가?
왜 다음 실험으로 넘어갔는가?
```

이런 기록은 팀 협업과 재현성에 필수입니다.

---

## Step 5. Evaluation

Evaluation은 최종 모델이 실제로 쓸 만한지 검증하는 단계입니다.

### 이 단계에서 해야 할 일

```text
hold-out test set으로 모델 평가
사전에 정한 error metric으로 성능 측정
모델 출력 해석
데이터 segment별 성능 확인
소프트웨어 unit test / integration test
모델 unit test
directional expectation test
alpha / beta user testing
```

---

## Directional Expectation Test란?

Directional expectation test는 입력을 특정 방향으로 바꿨을 때 모델 출력도 기대한 방향으로 바뀌는지 확인하는 테스트입니다.

예를 들어 전력 장애 예측 모델이라면, 다른 조건이 같을 때 폭풍 강도가 커지면 예상 정전 수가 증가하는 방향으로 나와야 자연스럽습니다. 이런 테스트는 모델이 완전히 이상한 방식으로 작동하지 않는지 확인하는 데 도움이 됩니다.

---

## Step 6. Deployment

Deployment는 모델을 실제 제품과 사용자 환경에 배포하는 단계입니다.

### 이 단계에서 해야 할 일

```text
모델 API 설계
소프트웨어 제품과 모델 통합
인프라 확장성 확보
보안 고려
일반 소프트웨어 배포 절차 수행
사용자 교육
성능 모니터링
재학습 일정 수립
```

ML 모델은 배포했다고 끝나는 것이 아닙니다. 환경이 변하면 모델 성능도 변할 수 있기 때문에, 운영 중 모니터링과 재학습 계획이 필수입니다.

# 5. CRISP-DM Case Study — Storm Outage Prediction

## 사례 개요

강의에서는 전력 회사가 폭풍이 오기 전에 정전의 심각도와 위치를 예측하는 ML 제품 사례를 설명합니다.

이 제품의 목적은 전력 회사가 폭풍 복구 인력을 더 잘 계획하도록 돕는 것이었습니다.

```text
문제 상황
→ 폭풍이 오기 2~3일 전에 필요한 복구 인력 규모를 결정해야 함

ML 제품
→ 폭풍으로 인한 정전 규모와 위치를 예측

기대 효과
→ 복구 시간 단축
→ 과도한 인력 배치로 인한 비용 낭비 감소
```

---

## Step 1. Business Understanding 적용

### 타깃 사용자

```text
전력 회사의 운영 책임자
Director of Operations
Manager of Operations
```

이들의 핵심 업무는 폭풍 이후 복구 작업을 계획하는 것입니다. 폭풍이 오기 전에 얼마나 많은 복구 인력을 준비해야 할지 결정해야 합니다.

---

## 사용자 문제

복구 인력을 너무 많이 준비하면 비용이 낭비됩니다. 인건비뿐 아니라 숙소, 식사 등 추가 비용도 발생합니다.

반대로 복구 인력을 너무 적게 준비하면 정전 복구가 늦어지고, 고객 불만이 커질 수 있습니다. 심한 경우 운영 책임자는 규제 기관이나 공공기관 앞에서 왜 충분한 인력을 준비하지 않았는지 설명해야 할 수도 있습니다.

```text
인력 과대 배치
→ 비용 낭비

인력 과소 배치
→ 복구 지연
→ 고객 불만
→ 규제 리스크
```

---

## 기존 해결 방식

기존에는 운영 책임자가 날씨 예보를 보고 자신의 경험과 직관을 활용해 필요한 인력 규모를 추정했습니다.

```text
날씨 예보
+ 운영자의 경험
+ 직관적 판단
→ 복구 인력 규모 결정
```

이 방식은 실무 경험을 활용한다는 장점은 있지만, 불확실성이 크고 일관된 판단을 내리기 어렵습니다.

---

## 성공 기준

이 프로젝트의 성공은 단순히 모델의 예측 성능이 좋아지는 것이 아니라, 실제 운영 결과가 좋아지는 것입니다.

| 구분 | 예시 |
| :--- | :--- |
| Outcome Metric | 평균 복구 시간 감소 |
| Output Metric | 정전 수 예측 모델의 MSE 감소 |
| Constraint | 폭풍 시작 최소 48시간 전에 예측 제공 |

여기서 중요한 점은 예측이 의사결정 시간 안에 제공되어야 한다는 것입니다. 아무리 정확한 예측이라도 폭풍이 이미 시작된 뒤에 나오면 운영 계획에 쓸 수 없습니다.

## Step 3. Data Preparation 적용

수집한 데이터에는 여러 문제가 있었습니다.

```text
결측치
서로 다른 지리적 해상도
이상치 폭풍
데이터 소스 간 기준 불일치
```

예를 들어 어떤 데이터는 5마일 x 5마일 격자 단위였고, 어떤 데이터는 도시 단위, 우편번호 단위였습니다. 따라서 서로 다른 데이터를 공통된 지리적 단위로 맞추는 작업이 필요했습니다.

또한 일부 폭풍은 다른 폭풍과 비교해 매우 큰 정전을 만들었습니다. 이런 outlier storm은 단순히 제거할 것이 아니라, 왜 그런 결과가 나왔는지 이해해야 했습니다.

## Feature Engineering 고려사항

이 사례에서는 날씨 feature만 해도 다양한 방식으로 만들 수 있었습니다.

```text
시간당 평균 풍속
하루 평균 풍속
시간당 최대 돌풍
특정 풍속 이상이 지속된 시간
강수량
눈 / 얼음 여부
최근 강우량
토양 포화 상태
```

특히 feature 간 상호작용도 중요했습니다. 예를 들어 최근 비가 많이 와서 땅이 젖어 있으면, 낮은 풍속에서도 나무가 더 쉽게 쓰러질 수 있습니다.

```text
최근 강우량 증가
→ 토양이 젖음
→ 나무가 쉽게 쓰러짐
→ 낮은 풍속에서도 정전 가능성 증가
```

## Step 4. Modeling 적용

이 프로젝트에서는 모델 성능뿐 아니라 해석 가능성도 중요했습니다.

복잡한 neural network를 사용할 수도 있었지만, 고객에게 설명 가능해야 하고, 모델이 틀렸을 때 디버깅할 수 있어야 했습니다. 그래서 더 단순하고 해석 가능한 모델을 선택했습니다.

```text
성능이 좋아야 함
+ 고객에게 설명 가능해야 함
+ 틀렸을 때 원인을 분석할 수 있어야 함
```

또 다른 중요한 판단은 하나의 공통 모델을 쓸지, 고객별 맞춤 모델을 만들지였습니다.

| 선택지 | 장점 | 단점 |
| :--- | :--- | :--- |
| 공통 모델 | 관리가 쉬움 | 각 지역 특성을 반영하기 어려움 |
| 고객별 모델 | 지역 특성 반영 가능 | 개발/운영 비용 증가 |

---

## Step 5. Evaluation 적용

이 사례에서는 두 가지 방식으로 모델을 테스트했습니다.

### 1. Leave-one-storm-out cross-validation

과거 폭풍 데이터 중 하나의 폭풍을 제외하고 나머지 폭풍으로 모델을 학습한 뒤, 제외했던 폭풍을 예측합니다. 이를 모든 폭풍에 대해 반복합니다.

```text
폭풍 A 제외
→ 나머지 폭풍으로 학습
→ 폭풍 A 예측

폭풍 B 제외
→ 나머지 폭풍으로 학습
→ 폭풍 B 예측

이 과정을 전체 폭풍 데이터에 반복
```

이 방식은 과거 데이터가 많지 않은 상황에서 모델이 새로운 폭풍에 얼마나 잘 일반화되는지 확인하는 데 유용합니다.

### 2. Live customer testing

모든 과거 데이터로 모델을 학습한 뒤, 실제 새로운 폭풍이 발생할 때 고객과 함께 모델 예측을 검증했습니다.

```text
과거 데이터 전체로 학습
→ 실제 폭풍 발생
→ live data 입력
→ 모델 예측
→ 실제 정전 결과와 비교
```

테스트 과정에서 데이터 품질 문제가 발견되었고, 팀은 다시 데이터 정제 단계로 돌아가 반복적으로 개선했습니다.

최종 모델은 독립된 모델로만 존재한 것이 아니라, 고객이 이미 사용하는 weather visualization product에 통합되었습니다.


## 배포 단계의 핵심: Change Management

이 제품은 고객의 기존 업무 방식을 바꾸는 제품이었습니다. 고객은 원래 날씨 예보와 경험에 의존해 인력 규모를 정했지만, 이제는 ML 예측 도구를 함께 활용해야 했습니다.

따라서 배포에는 단순 기술 통합뿐 아니라 사용자의 업무 변화 관리가 필요했습니다.

```text
사용자가 모델 결과를 이해하도록 돕기
기존 업무 흐름에 모델을 자연스럽게 통합하기
모델을 신뢰하도록 교육하기
잘못된 예측이 나왔을 때 설명할 수 있게 하기
```

## 배포 이후 모니터링과 재학습

모델은 한 번 배포하고 끝낼 수 없습니다. 시간이 지나면 환경이 변합니다.

이 사례에서는 다음과 같은 변화가 발생할 수 있습니다.

```text
전력선 주변 나무가 가지치기됨
새로운 나무가 심어짐
폭풍으로 기존 나무가 쓰러짐
전력 회사가 새로운 설비를 추가함
전력선 위치나 밀도가 바뀜
```

이런 변화가 생기면 과거 데이터로 학습한 모델의 성능이 떨어질 수 있습니다. 따라서 모델 성능을 계속 모니터링하고, 새 데이터를 반영해 주기적으로 재학습해야 합니다.

# 6. 실무적으로 중요한 인사이트

## 1. ML 프로젝트는 모델링보다 문제 정의가 먼저다

좋은 모델을 만드는 것보다 먼저 해야 할 일은 올바른 문제를 정의하는 것입니다. 문제 정의가 잘못되면 이후 데이터 수집, 모델링, 평가가 모두 잘못된 방향으로 갈 수 있습니다.

---

## 2. Output Metric과 Outcome Metric을 구분해야 한다

모델의 MSE가 낮아지는 것은 output 성능입니다. 하지만 고객이 진짜 원하는 것은 복구 시간 감소, 비용 절감, 의사결정 개선 같은 outcome입니다.

```text
Output이 좋아짐
≠ 반드시 비즈니스 Outcome이 좋아짐
```

좋은 ML 프로젝트는 output metric과 outcome metric을 함께 관리해야 합니다.

---

## 3. ML 프로젝트는 반복을 전제로 계획해야 한다

ML 프로젝트는 한 번에 깔끔하게 진행되지 않습니다. 모델 성능이 안 나오면 데이터 단계로 돌아가야 하고, 사용자 테스트에서 문제가 발견되면 문제 정의 자체를 수정해야 할 수도 있습니다.

---

## 4. 데이터 품질이 모델 성능을 지배한다

모델 알고리즘보다 먼저 봐야 할 것은 데이터입니다. 결측치, 이상치, 잘못된 label, 서로 다른 기준의 데이터가 해결되지 않으면 모델 성능은 좋아지기 어렵습니다.

---

## 5. 해석 가능성은 성능만큼 중요할 수 있다

항상 가장 복잡한 모델이 좋은 선택은 아닙니다. 고객에게 설명해야 하거나, 오류 발생 시 원인을 파악해야 하는 상황에서는 해석 가능한 모델이 더 적합할 수 있습니다.

---

## 6. 배포는 기술 문제가 아니라 업무 변화 문제이기도 하다

ML 제품이 사용자의 workflow를 바꾸면, 사용자는 자연스럽게 저항하거나 불신할 수 있습니다. 따라서 모델을 배포할 때는 사용자 교육, 신뢰 형성, 설명 가능성, 변화 관리까지 함께 설계해야 합니다.

---

# 7. 프로젝트 / 면접에 써먹을 수 있는 문장

ML 프로젝트는 일반 소프트웨어 프로젝트와 달리 데이터 품질, 모델 성능, 환경 변화, 사용자 신뢰까지 함께 관리해야 합니다. 따라서 CRISP-DM과 같은 반복적 프로세스를 통해 문제 정의, 데이터 이해, 데이터 준비, 모델링, 평가, 배포를 구조화하고, output metric과 outcome metric을 함께 관리하는 것이 중요합니다.

# 최종 정리

Module 2의 핵심은 ML 프로젝트를 단순히 모델을 만드는 일로 보면 안 된다는 것입니다. ML 프로젝트는 문제 정의, 데이터 품질, 모델 실험, 평가, 배포, 운영, 사용자 변화 관리까지 포함하는 복합적인 프로젝트입니다.

CRISP-DM은 이 복잡한 과정을 관리하기 위한 실용적인 틀입니다. 중요한 것은 단계를 외우는 것이 아니라, 왜 이 순서가 필요한지 이해하는 것입니다. 문제를 잘못 정의하면 모델이 아무리 좋아도 쓸모없고, 데이터를 제대로 준비하지 않으면 모델링은 실패하며, 배포 후 모니터링과 재학습이 없으면 모델은 현실 변화에 뒤처집니다.
