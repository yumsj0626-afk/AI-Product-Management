
# Managing Machine Learning Projects

## Module 3 — Data Considerations

### Sourcing Data: Data Needs / Data Collection

---

## 한 줄 요약

ML 프로젝트에서 데이터는 단순히 “많이 모으면 좋은 것”이 아니라, **모델 학습에 필요한 과거 데이터와 실제 예측에 필요한 실시간 입력 데이터를 구분해서 설계해야 하는 자원**입니다. 좋은 데이터 전략은 필요한 feature와 label을 정의하고, 대표성·품질·비용·프라이버시·지속적 업데이트까지 함께 고려하는 것입니다.

---

# 1. Data Needs

## 핵심 개념

ML 프로젝트에서 데이터는 크게 두 가지 목적에 필요합니다.

```text
1. Training
   → 모델을 학습시키기 위한 과거 데이터

2. Prediction / Inference
   → 학습된 모델이 실제 예측을 수행하기 위한 현재 또는 실시간 데이터
```

즉, 모델을 만들 때 필요한 데이터와 모델을 운영할 때 필요한 데이터는 다릅니다.

---

## Training Data와 Prediction Data

| 구분 | 필요한 데이터 | 목적 |
| :--- | :--- | :--- |
| Training Data | 과거 input features + 과거 target/label | 모델이 패턴을 학습하도록 함 |
| Prediction Data | 현재 또는 실시간 input features | 학습된 모델이 새로운 예측을 생성하도록 함 |

예를 들어 전력 수요 예측 모델을 만든다면, 학습 단계에서는 과거의 날씨, 요일, 시간대, 실제 전력 사용량이 필요합니다. 하지만 운영 단계에서는 내일의 날씨 예보, 요일, 시간대 같은 입력값이 계속 들어와야 실제 예측을 만들 수 있습니다.

```text
Historical Features + Historical Labels
→ Model Training
→ Trained Model

Real-time / New Features
→ Trained Model
→ Prediction
```

---

# 2. Feature를 어떻게 찾을 것인가?

Feature는 모델이 예측을 위해 사용하는 입력 변수입니다. 어떤 feature를 사용할지 정하는 것은 모델 성능에 큰 영향을 줍니다.

강의에서는 feature 후보를 찾는 주요 출처를 세 가지로 설명합니다.

---

## 1. Subject Matter Experts

가장 좋은 출발점은 문제를 잘 아는 도메인 전문가에게 묻는 것입니다.

자신이 해당 문제의 전문가가 아니라면, 실제 현상을 잘 이해하는 사람들과 이야기해야 합니다. 도메인 전문가는 어떤 요인이 문제에 영향을 줄 가능성이 높은지 알려줄 수 있습니다.

```text
문제 이해
→ 도메인 전문가 인터뷰
→ 중요한 요인 후보 도출
→ feature 후보 생성
```

---

## 2. Customers

고객도 중요한 feature 후보의 출처입니다. 고객은 실제 문제를 겪고 있는 사람이기 때문에, 어떤 요인이 문제를 악화시키거나 완화시키는지 경험적으로 알고 있을 수 있습니다.

특히 고객과의 대화는 단순히 feature를 찾는 것뿐 아니라, 문제 자체를 더 정확히 정의하는 데도 도움이 됩니다.

---

## 3. Temporal / Geospatial Characteristics

문제에 시간이나 위치 요소가 있다면, 이 특성을 feature로 고려해야 합니다.

| 특성 | Feature 예시 |
| :--- | :--- |
| 시간 특성 | 시간대, 요일, 월, 계절, 공휴일 여부 |
| 시계열 특성 | 이전 값, 이동평균, 최근 변화량 |
| 위치 특성 | 지역, 거리, 밀도, 지리적 구역 |
| 계절성 | 여름/겨울, 성수기/비수기, 특정 시즌 |

예를 들어 제품 수요를 예측한다면 요일, 계절, 시간대가 중요할 수 있습니다. 특정 지역의 사고나 고장을 예측한다면 위치, 밀도, 주변 환경 같은 정보가 중요할 수 있습니다.

---

# 3. Feature는 얼마나 많이 필요한가?

정답은 문제에 따라 다릅니다. 하지만 일반적으로는 **작은 feature set으로 baseline을 먼저 만들고**, 이후 feature를 추가하면서 성능 개선 여부를 확인하는 방식이 좋습니다.

```text
작은 feature set으로 시작
→ baseline model 생성
→ feature 추가
→ 성능 변화 확인
→ 도움이 되면 유지, 아니면 제거
```

강의에서 중요한 조언은 이것입니다.

> 중요한 feature를 빠뜨리는 것이, 불필요한 feature를 조금 더 넣는 것보다 훨씬 더 위험하다.

즉, 논리적으로 중요해 보이는 feature라면 일단 시도해보고, 성능에 도움이 되는지 검증하는 것이 좋습니다.

---

## Feature 선택의 실무 감각

| 상황 | 접근 |
| :--- | :--- |
| 처음 시작할 때 | 핵심 feature만 사용해 baseline 구성 |
| 도메인상 중요해 보이는 요인이 있을 때 | 일단 feature로 만들어 실험 |
| 성능이 개선될 때 | 유지 |
| 성능 개선이 없거나 노이즈가 클 때 | 제거 또는 재설계 |
| feature 수가 너무 많을 때 | feature selection 또는 차원 축소 고려 |

---

# 4. Label / Target이란?

Feature와 함께 필요한 것이 label 또는 target입니다.

Label은 모델이 맞혀야 하는 정답입니다. 문제 정의에 따라 label의 형태도 달라집니다.

---

## 문제 유형에 따른 Label 예시

| 문제 유형 | Label 예시 |
| :--- | :--- |
| 판매량 예측 | 실제 판매량 |
| 이미지 분류 | 이미지 안의 대표 객체 이름 |
| 객체 탐지 | 객체 종류 + 위치 좌표 |
| 질병 진단 | 질병 여부 또는 질병 종류 |
| 추천 시스템 | 클릭 여부, 구매 여부, 평점 |
| 정전 예측 | 실제 정전 수 또는 정전 위치 |

예를 들어 이미지 분류 문제라면 label은 “고양이”, “강아지”, “자동차” 같은 클래스일 수 있습니다. 하지만 객체 탐지 문제라면 단순히 이미지의 대표 객체만 알면 부족하고, 이미지 안의 모든 객체와 그 위치까지 label로 필요합니다.

```text
Image Classification
→ 이미지 전체에 대한 class label 필요

Object Detection
→ 객체 class + bounding box 위치 필요
```

---

## Label은 직접 만들어야 할 수도 있다

Feature 데이터는 존재하더라도 label이 없는 경우가 많습니다. 이런 경우에는 사람이 직접 label을 만들어야 합니다.

예를 들어 이미지 파일은 많이 있어도, 그 이미지 안에 무엇이 있는지 정답이 없다면 사람이 직접 “이 이미지는 고양이”, “이 이미지는 자동차”처럼 라벨링해야 합니다.

이 과정은 비용과 시간이 많이 들 수 있기 때문에, ML 프로젝트 초기에 label 확보 가능성을 반드시 확인해야 합니다.

---

# 5. 얼마나 많은 데이터가 필요한가?

일반적으로 데이터는 많을수록 좋습니다. 하지만 필요한 데이터 양은 문제의 복잡도에 따라 달라집니다.

강의에서는 보통 feature 수보다 훨씬 많은 observation이 필요하다고 설명합니다. 특히 feature가 많거나 입력과 target 사이의 관계가 복잡할수록 훨씬 많은 데이터가 필요합니다.

---

## 데이터 요구량을 결정하는 요인

| 요인 | 설명 |
| :--- | :--- |
| Feature 수 | feature가 많을수록 더 많은 observation 필요 |
| 관계의 복잡도 | 비선형적이고 복잡한 관계일수록 더 많은 데이터 필요 |
| 데이터 품질 | 결측치나 노이즈가 많으면 더 많은 데이터가 필요할 수 있음 |
| 목표 성능 수준 | 높은 성능을 원할수록 더 많은 데이터가 필요할 가능성이 큼 |
| 문제 유형 | 이미지, 텍스트, 자연어처럼 복잡한 문제는 대량 데이터 필요 |

---

## 데이터 양에 대한 직관

| 예시 | 데이터 규모 | 이유 |
| :--- | :--- | :--- |
| Iris flower dataset | 약 150개 observation | feature가 4개이고 관계가 비교적 단순함 |
| Chest X-ray dataset | 약 224,000개 이미지 | 의료 이미지 분류는 feature가 많고 복잡함 |
| ImageNet | 약 1,400만 개 이미지 | 수천 개 객체 클래스를 다루는 대규모 이미지 문제 |
| Gmail Smart Reply | 약 2억 3,800만 개 observation | 자연어 응답 생성은 복잡한 패턴을 다룸 |
| Google Translate | 수조 개 observation | 다국어 번역은 feature와 관계가 매우 복잡함 |

핵심은 단순한 문제는 적은 데이터로도 높은 성능을 낼 수 있지만, 이미지·텍스트·언어처럼 복잡한 문제는 훨씬 많은 데이터가 필요하다는 것입니다.

---

# 6. Data Collection

## 핵심 개념

데이터는 다양한 곳에서 얻을 수 있습니다. 하지만 중요한 것은 가능한 모든 데이터를 무작정 모으는 것이 아니라, **모델에 실제로 필요하다고 판단되는 데이터를 의도적으로 수집하는 것**입니다.

---

## 데이터 소스의 종류

| 데이터 소스 | 예시 |
| :--- | :--- |
| Internal Data | 로그 파일, 웹사이트 사용자 데이터, 운영 데이터, ERP 시스템 데이터 |
| Customer Data | 고객사의 센서 데이터, 운영 시스템 데이터, 하드웨어 데이터 |
| User Data | 웹사이트 행동 로그, 투표, 평점, 리뷰, 설문/폼 입력 |
| External Third-party Data | 날씨 데이터, 인구통계 데이터, 소셜미디어 데이터, 외부 vendor 데이터 |

---

## Internal Data

Internal Data는 조직 내부에서 이미 생성되고 있는 데이터입니다.

예를 들어 웹 서비스를 운영한다면 사용자 로그, 클릭 기록, 검색 기록, 구매 기록이 있을 수 있습니다. 제조업이나 하드웨어 기업이라면 장비 운영 데이터나 센서 데이터가 있을 수 있습니다. ERP 시스템도 중요한 내부 데이터 소스가 될 수 있습니다.

---

## Customer Data

고객사로부터 데이터를 수집할 수도 있습니다.

예를 들어 고객의 공장 장비에 센서를 설치하거나, 고객의 운영 시스템 또는 하드웨어에서 데이터를 가져올 수 있습니다. B2B ML 프로젝트에서는 고객 데이터가 매우 중요하지만, 접근 권한, 민감도, 보안, 데이터 품질 문제가 함께 따라옵니다.

---

## User Data

웹 서비스나 앱에서는 사용자의 행동 자체가 데이터가 됩니다.

```text
회원가입 폼
검색 기록
클릭 로그
구매 기록
평점
리뷰
좋아요 / 싫어요
추천 결과에 대한 반응
```

이런 데이터는 개인화, 추천, 검색 최적화, 사용자 행동 예측 등에 활용될 수 있습니다.

---

## External Third-party Data

외부 데이터도 모델링에 중요할 수 있습니다.

예를 들어 날씨 데이터는 수요 예측, 물류, 에너지, 농업, 보험 등 다양한 분야에서 유용할 수 있습니다. 인구통계 데이터는 특정 사용자 집단을 이해하는 데 도움이 되고, 소셜미디어 데이터는 트렌드나 여론을 분석하는 데 쓰일 수 있습니다.

---

# 7. Data Collection Best Practices

## 1. Collect Data Intentionally

데이터는 의도적으로 수집해야 합니다. “있으면 언젠가 쓰겠지”라는 생각으로 모든 데이터를 모으는 것은 좋은 전략이 아닙니다.

데이터를 많이 모을수록 저장 비용, 처리 비용, 보안 리스크, 프라이버시 문제가 함께 증가합니다.

```text
문제 정의
→ 필요한 feature 정의
→ 필요한 label 정의
→ 필요한 데이터 소스 선정
→ 목적에 맞게 수집
```

---

## 무작정 데이터를 모으면 생기는 문제

| 문제 | 설명 |
| :--- | :--- |
| 저장 비용 증가 | 수집한 데이터를 보관해야 함 |
| 처리 비용 증가 | 큰 데이터를 처리하는 데 비용과 시간이 듦 |
| 프라이버시 리스크 | 민감한 정보를 과도하게 수집할 수 있음 |
| 윤리 문제 | 사용자 동의나 목적 외 사용 문제가 발생할 수 있음 |
| 관리 복잡도 증가 | 데이터 출처와 의미를 추적하기 어려워짐 |

---

## 2. Avoid Bias

데이터를 수집할 때 편향이 들어가지 않도록 조심해야 합니다.

데이터가 특정 사용자 집단, 특정 지역, 특정 시간대, 특정 상황에만 치우쳐 있으면 모델도 그 편향을 학습합니다.

```text
편향된 데이터
→ 편향된 모델
→ 특정 집단이나 상황에서 낮은 성능
→ 잘못된 의사결정
```

---

## 3. Collect Representative Data

수집한 데이터는 모델이 실제로 적용될 population을 대표해야 합니다.

예를 들어 전체 사용자를 위한 모델을 만들면서 특정 지역, 특정 연령대, 특정 행동 패턴을 가진 사용자 데이터만 모으면 실제 운영 환경에서 성능이 떨어질 수 있습니다.

| 나쁜 데이터 수집 | 문제 |
| :--- | :--- |
| 일부 사용자 집단만 수집 | 전체 사용자에 일반화하기 어려움 |
| 특정 시간대 데이터만 수집 | 시간 변화 반영 어려움 |
| 특정 지역 데이터만 수집 | 다른 지역 성능 저하 가능 |
| 성공 사례만 수집 | 실패 패턴을 학습하지 못함 |

---

## 4. Keep Updating Data

현실 세계는 계속 변합니다. 사용자의 행동, 시장 상황, 장비 상태, 환경 조건이 변하면 과거 데이터로 학습한 모델의 성능이 점점 떨어질 수 있습니다.

따라서 데이터 수집은 한 번 하고 끝나는 작업이 아니라, 지속적으로 이루어져야 합니다.

```text
새 데이터 수집
→ 성능 모니터링
→ 데이터 분포 변화 감지
→ 필요 시 재학습
→ 모델 업데이트
```

---

## 5. Document Data Sources and Metadata

데이터를 수집할 때는 반드시 문서화해야 합니다.

나중에 팀원이 바뀌거나 모델 성능 문제가 발생했을 때, 데이터가 어디서 왔고 어떤 의미를 가지며 어떤 처리 과정을 거쳤는지 모르면 큰 문제가 됩니다.

---

## 문서화해야 할 내용

| 항목 | 설명 |
| :--- | :--- |
| Data Source | 데이터가 어디서 왔는가 |
| Collection Method | 어떻게 수집했는가 |
| Time Range | 어느 기간의 데이터인가 |
| Feature Definition | 각 컬럼이 무엇을 의미하는가 |
| Label Definition | target/label이 어떻게 정의되었는가 |
| Data Quality Issues | 결측치, 오류, 이상치가 있었는가 |
| Processing History | 어떤 정제와 변환을 했는가 |
| Usage Constraints | 개인정보, 보안, 라이선스 제한이 있는가 |

---

# 8. User Data Collection

## 핵심 개념

사용자 데이터는 ML 제품에서 매우 강력한 데이터 소스입니다. 하지만 사용자에게 부담을 주지 않는 방식으로 수집하는 것이 중요합니다.

가장 좋은 방식은 사용자의 자연스러운 workflow 안에서 데이터를 수집하는 것입니다.

```text
나쁜 방식
→ 사용자가 귀찮게 별도 입력해야 함

좋은 방식
→ 사용자가 원래 하던 행동이 자연스럽게 데이터가 됨
```

---

## 좋은 사용자 데이터 수집의 조건

| 조건 | 설명 |
| :--- | :--- |
| Non-obtrusive | 사용자에게 방해가 적어야 함 |
| Workflow-integrated | 사용자가 원래 하던 행동 안에서 수집되어야 함 |
| User Benefit | 사용자가 데이터를 제공함으로써 이득을 느껴야 함 |
| Continuous | 사용자가 계속 사용할수록 데이터가 쌓여야 함 |

---

## 예시 1. CAPTCHA

CAPTCHA는 사용자가 봇이 아니라 사람인지 확인하는 기능입니다. 하지만 동시에 이미지 라벨링 데이터 수집에도 활용될 수 있습니다.

예를 들어 “횡단보도가 있는 이미지를 모두 선택하세요”라는 CAPTCHA는 사용자가 사람인지 검증하는 동시에, 횡단보도 이미지에 대한 label을 제공하게 만듭니다.

```text
사용자 입장
→ 사람임을 인증

시스템 입장
→ 이미지 라벨 데이터 수집
```

---

## 예시 2. Stitch Fix

Stitch Fix는 사용자에게 옷 박스를 보내고, 사용자는 마음에 드는 옷은 구매하고 마음에 들지 않는 옷은 반품합니다.

이 과정 자체가 사용자의 선호 데이터를 만듭니다.

```text
사용자가 옷을 keep
→ 선호하는 아이템으로 label

사용자가 옷을 return
→ 선호하지 않는 아이템으로 label
```

사용자는 단순히 옷을 고르는 행동을 하지만, 시스템은 이 행동을 이용해 개인화 추천 모델을 개선할 수 있습니다.

---

# 9. Flywheel Effect

## 핵심 개념

Flywheel Effect는 사용자가 제품을 사용할수록 데이터가 쌓이고, 그 데이터가 AI 시스템을 더 좋게 만들고, 더 좋은 시스템이 다시 더 많은 사용과 데이터를 만들어내는 순환 구조입니다.

```text
사용자 사용 증가
→ 데이터 증가
→ 모델 성능 개선
→ 사용자 경험 개선
→ 더 많은 사용
→ 더 많은 데이터
```

이 구조가 만들어지면 ML 제품은 시간이 갈수록 강해질 수 있습니다.

---

## Amazon 예시

Amazon은 사용자의 검색, 구매, 평점 데이터를 활용해 여러 방식으로 모델을 개선합니다.

| 사용자 데이터 | 활용 방식 |
| :--- | :--- |
| 검색 후 구매 기록 | 검색 결과 순서 개선 |
| 개인 구매/평점 기록 | 개인화 추천 |
| 함께 구매된 상품 기록 | “함께 구매한 상품” 추천 |

예를 들어 많은 사용자가 손전등을 검색한 뒤 특정 손전등을 많이 구매했다면, Amazon은 그 상품을 검색 결과 상단에 배치할 수 있습니다.

또한 손전등을 산 사람들이 배터리도 함께 샀다면, Amazon은 “다른 고객이 함께 구매한 상품”으로 배터리를 추천할 수 있습니다. 이런 방식은 함께 등장하는 항목을 분석하는 co-occurrence matrix와 연결됩니다.

---

# 10. Cold Start Problem

## 핵심 개념

Cold Start Problem은 새로운 사용자나 새로운 아이템에 대해 데이터가 부족해 개인화된 예측이나 추천을 하기 어려운 문제입니다.

추천 시스템에서 특히 자주 발생합니다.

```text
새로운 사용자 가입
→ 아직 시청/구매/평점 기록 없음
→ 선호를 알 수 없음
→ 개인화 추천 어려움
```

---

## Cold Start 해결 방식

| 방식 | 설명 |
| :--- | :--- |
| Heuristic 사용 | 인기 상품, 최고 평점 상품 등 단순 기준으로 추천 |
| Calibration Step 추가 | 가입 초기에 몇 가지 선호 정보를 직접 입력받음 |
| 초기 행동 데이터 활용 | 첫 클릭, 첫 검색, 첫 구매를 빠르게 반영 |
| 시간이 지나며 재학습 | 사용자 행동이 쌓이면 모델을 점점 개선 |

---

## Netflix 예시

Netflix에 새로운 사용자가 가입하면 아직 본 영화나 평점 데이터가 없기 때문에 개인화 추천을 하기 어렵습니다.

이때 사용할 수 있는 방법은 두 가지입니다.

```text
1. Heuristic Approach
   → 전체 사용자에게 인기 있거나 평점 높은 영화를 추천

2. Calibration Step
   → 가입 시 몇 개의 영화를 평가하게 함
   → 초기 선호를 파악
   → 더 나은 추천 제공
```

즉, 데이터가 없는 초반에는 단순한 규칙이나 초기 선호 수집 단계를 활용하고, 사용자가 서비스를 이용하면서 생기는 데이터를 바탕으로 점점 모델을 개선합니다.

---

# 11. 실무적으로 중요한 인사이트

## 1. 데이터는 학습용과 운영용을 나눠서 생각해야 한다

모델 학습에는 과거 feature와 label이 필요하고, 실제 운영에는 현재 또는 실시간 feature가 필요합니다. 학습 데이터만 있고 운영 시점에 feature를 얻을 수 없다면 모델은 실제로 쓸 수 없습니다.

---

## 2. Feature는 도메인 이해에서 시작한다

좋은 feature는 단순히 데이터셋 안에서 자동으로 나오는 것이 아닙니다. 도메인 전문가, 고객, 시간/공간 특성 이해를 통해 만들어집니다.

---

## 3. Label 확보 가능성을 초기에 확인해야 한다

많은 ML 프로젝트는 feature보다 label 확보에서 막힙니다. 정답 데이터가 없으면 지도학습 모델을 만들 수 없고, label을 직접 만들어야 한다면 비용과 시간이 크게 늘어납니다.

---

## 4. 데이터는 많을수록 좋지만, 무작정 많이 모으면 안 된다

데이터가 많으면 모델 성능에 도움이 될 수 있지만, 저장 비용, 처리 비용, 프라이버시, 윤리 문제가 함께 커집니다. 따라서 필요한 데이터를 의도적으로 수집해야 합니다.

---

## 5. 대표성 없는 데이터는 위험하다

데이터가 실제 사용자를 대표하지 못하면 모델은 특정 상황에서만 잘 작동합니다. 운영 환경에 배포하면 성능이 급격히 떨어질 수 있습니다.

---

## 6. 사용자 데이터 수집은 제품 경험 안에 녹아 있어야 한다

사용자가 귀찮게 느끼는 방식의 데이터 수집은 지속되기 어렵습니다. 좋은 데이터 수집은 사용자의 자연스러운 행동이 곧 모델 개선 데이터가 되도록 설계하는 것입니다.

---

## 7. Cold Start는 추천 시스템의 핵심 문제다

새로운 사용자에게는 데이터가 없기 때문에 개인화 모델이 바로 작동하기 어렵습니다. 이때는 인기 상품 추천 같은 휴리스틱이나 초기 선호 수집 단계를 활용해야 합니다.

---

# 12. 프로젝트 / 면접에 써먹을 수 있는 문장

ML 프로젝트에서 데이터 요구사항은 학습에 필요한 historical features와 labels뿐 아니라, 실제 운영 시점에 예측을 생성하기 위한 real-time input까지 포함해야 합니다. 또한 데이터 수집은 비용, 프라이버시, 편향, 대표성, 지속적 업데이트를 고려해 의도적으로 설계해야 하며, 사용자 행동을 자연스럽게 데이터로 전환하는 구조를 만들면 flywheel effect를 통해 모델과 제품이 함께 개선될 수 있습니다.

---

# 최종 정리

Module 3의 핵심은 ML 프로젝트에서 데이터가 단순한 원재료가 아니라는 점입니다. 데이터는 문제 정의, feature 설계, label 확보, 모델 학습, 실시간 예측, 운영 후 재학습까지 전 과정에 영향을 줍니다.

좋은 데이터 전략은 “어떤 데이터를 많이 모을까?”가 아니라 “이 문제를 풀기 위해 어떤 feature와 label이 필요하고, 그 데이터를 어떻게 안정적이고 윤리적으로 수집·관리·업데이트할 것인가?”를 묻는 데서 시작합니다.
