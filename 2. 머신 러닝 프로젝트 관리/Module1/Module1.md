
# Managing Machine Learning Projects

## Module 1 — Identifying Opportunities for Machine Learning

### Why Use ML in Products? / Do You Really Need ML?

---

## 한 줄 요약

머신러닝은 “멋있어서” 쓰는 기술이 아니라, **사용자에게 명확한 비즈니스 가치를 만들 때만 도입해야 하는 도구**입니다. ML을 적용하기 전에는 먼저 간단한 휴리스틱 baseline을 만들고, ML이 그보다 충분히 나은 성능과 가치를 제공하는지 검증해야 합니다.

---

## 핵심 개념 정리

### 1. ML을 써야 하는 이유

머신러닝은 복잡하고 기술적 리스크가 큽니다. 따라서 제품에 ML을 넣을 때는 “AI가 유행이라서”가 아니라, 실제로 사용자 문제를 더 잘 해결하는지 따져야 합니다.

| 구분 | 설명 |
| :--- | :--- |
| 나쁜 이유 1 | 임원이나 매니저가 AI를 넣으라고 했기 때문 |
| 나쁜 이유 2 | ML이 멋있고 유행하는 기술이기 때문 |
| 좋은 이유 | 사용자에게 실질적인 비즈니스 가치를 만들기 때문 |

핵심은 **ML 도입 자체가 목적이 되면 안 된다**는 것입니다. ML은 제품 가치를 높이기 위한 수단이어야 합니다.

---

## ML이 비즈니스 가치를 만드는 3가지 방식

ML은 제품에서 주로 다음 세 가지 방식으로 가치를 만듭니다.

```text
1. Automation       → 반복 작업 자동화
2. Prediction       → 더 나은 의사결정을 위한 예측
3. Personalization  → 사용자별 맞춤 경험 제공
```

---

## 1. Automation — 반복 작업 자동화

Automation은 사용자가 하던 반복적이고 지루한 일을 기계가 대신 처리하게 만드는 것입니다. 이를 통해 비용을 줄이거나 품질을 개선할 수 있습니다.

예를 들어 피자 체인에서 오븐에서 나온 피자를 카메라와 컴퓨터 비전 시스템으로 검사하는 사례가 나옵니다. 피자에 올바른 재료가 들어갔는지, 치즈와 소스 비율이 적절한지 확인해서 품질 관리를 자동화하는 방식입니다.

### Automation vs Augmentation

| 구분 | 의미 | 예시 |
| :--- | :--- | :--- |
| Automation | 사람이 하던 일을 기계가 대체 | 피자 품질 검사 자동화 |
| Augmentation | 사람이 더 잘 판단하도록 기계가 보조 | 폐렴 진단에서 방사선 전문의를 ML이 보조 |

모든 일을 완전히 자동화하는 것이 항상 좋은 선택은 아닙니다. 의료처럼 책임이 크고 위험도가 높은 분야에서는 인간을 완전히 제거하기보다, ML이 전문가를 보조하는 방식이 더 적절할 수 있습니다.

### Automation의 주요 리스크

| 리스크 | 설명 |
| :--- | :--- |
| 환경 변화 | 주변 환경이 바뀌면 모델 성능이 급격히 떨어질 수 있음 |
| 윤리 문제 | ML 알고리즘은 인간처럼 윤리적 판단 기준을 스스로 갖고 있지 않음 |
| 책임 소재 | 모델이 잘못 판단했을 때 누가 책임지는지 불명확할 수 있음 |

---

## 2. Prediction — 더 나은 의사결정을 위한 예측

ML은 대량의 데이터에서 사람이 찾기 어려운 패턴을 찾아 예측하는 데 강합니다. 이 예측은 사용자의 의사결정을 지원할 수 있습니다.

예를 들어 전력 회사는 다음 날 전력 수요를 예측해야 합니다. 과거 사용량 데이터와 여러 feature를 활용해 수요를 예측하면, 발전 계획을 더 잘 세울 수 있습니다.

```text
과거 데이터 + feature
→ 패턴 학습
→ 미래 값 예측
→ 더 나은 의사결정 지원
```

### Prediction의 주요 리스크

| 리스크 | 설명 |
| :--- | :--- |
| 환경 변화 | 세상이 바뀌면 과거 데이터로 학습한 모델이 맞지 않을 수 있음 |
| 높은 오답 비용 | 예측이 틀렸을 때 피해가 크면 ML 적용을 신중히 해야 함 |
| 모델 관리 필요 | 배포 후에도 성능을 평가하고 필요하면 조정해야 함 |

예를 들어 채용 후보자의 성공 가능성을 예측하는 모델은 매우 조심해야 합니다. 예측이 틀렸을 때 회사와 지원자 모두에게 큰 피해가 갈 수 있기 때문입니다.

---

## 3. Personalization — 사용자별 맞춤 경험 제공

ML은 사용자 데이터를 바탕으로 각 사용자의 선호, 지식 수준, 행동 패턴을 학습할 수 있습니다. 이를 통해 더 개인화된 경험을 제공할 수 있습니다.

대표적인 예시는 Netflix의 영화 추천 시스템입니다. 사용자의 시청 기록과 평가 데이터를 바탕으로 각 사용자가 좋아할 만한 영화를 추천합니다.

교육 분야에서도 비슷하게 활용할 수 있습니다. 학생들의 지식 상태를 파악해 각 학생에게 다른 예제, 과제, 학습 경로를 제공하는 방식입니다.

```text
사용자 데이터 수집
→ 선호 / 지식 상태 / 행동 패턴 학습
→ 사용자별 맞춤 추천 또는 경험 제공
```

# Do You Really Need ML?

## 핵심 질문

ML 프로젝트를 시작할 때 가장 먼저 물어야 할 질문은 이것입니다.

> 이 문제를 해결하는 데 정말 머신러닝이 필요한가?

ML은 강력하지만 데이터 준비, 모델 훈련, 운영, 모니터링, 재학습 비용이 들어갑니다. 따라서 간단한 규칙으로 충분히 해결되는 문제라면 굳이 ML을 적용하지 않아도 됩니다.

---

## 휴리스틱이란?

휴리스틱은 과거 경험이나 데이터를 바탕으로 만든 **간단한 규칙 기반 해결 방법**입니다. 복잡한 모델을 만들기 전에 시도하는 “상식적인 기준선”이라고 볼 수 있습니다.

### 휴리스틱 예시

| 문제 | 휴리스틱 접근 |
| :--- | :--- |
| 일별 판매량 예측 | 전체 평균 판매량을 예측값으로 사용 |
| 금요일 판매량 예측 | 과거 금요일 평균 판매량 사용 |
| 이미지 분류 | 훈련 데이터에서 가장 흔한 클래스로 전부 예측 |
| 추천 시스템 | 모든 사용자에게 가장 평점이 높은 상품 추천 |

휴리스틱은 단순하지만 중요합니다. ML 모델이 정말 가치 있는지 판단할 수 있는 비교 기준이 되기 때문입니다.

---

## 휴리스틱 vs 머신러닝

| 항목 | 휴리스틱 | 머신러닝 |
| :--- | :--- | :--- |
| 방식 | 사람이 만든 규칙 기반 | 데이터에서 패턴 학습 |
| 구현 난이도 | 낮음 | 상대적으로 높음 |
| 유지보수 | 쉬움 | 모델 관리와 재학습 필요 |
| 계산 비용 | 낮음 | 훈련/추론 비용 발생 |
| 설명 가능성 | 높음 | 모델에 따라 낮을 수 있음 |
| 성능 | 복잡한 문제에서는 한계 | 충분한 데이터가 있으면 더 높은 성능 가능 |
| 환경 변화 대응 | 사람이 규칙을 직접 수정 | 새 데이터로 재학습 가능 |
| 적합한 문제 | 단순하고 규칙화 가능한 문제 | 데이터가 많고 패턴이 복잡한 문제 |

---

## 휴리스틱의 장점

휴리스틱의 장점은 명확합니다.

```text
쉽다
싸다
빠르다
설명하기 좋다
유지보수가 쉽다
```

예를 들어 “금요일 판매량은 과거 금요일 평균으로 예측했습니다”라고 하면 누구나 이해할 수 있습니다. 반면 복잡한 ML 모델은 왜 그런 예측을 했는지 설명하기 어려울 수 있습니다.

---

## 머신러닝이 더 적합한 경우

ML은 다음과 같은 상황에서 휴리스틱보다 유리할 수 있습니다.

| 상황 | 이유 |
| :--- | :--- |
| 데이터 양이 매우 많음 | 사람이 직접 규칙을 만들기 어려움 |
| feature가 많고 복잡함 | 변수 간 관계를 사람이 파악하기 어려움 |
| 이미지 / 음성 / 텍스트 처리 | 단순 규칙으로 처리하기 어려움 |
| 사용자별 패턴이 다름 | 개인화된 예측이나 추천이 필요함 |
| 환경 변화에 맞춰 재학습 필요 | 새 데이터를 반영해 모델을 갱신할 수 있음 |

특히 컴퓨터 비전처럼 이미지 안의 객체를 인식하거나 위치를 찾는 문제는 단순한 휴리스틱으로 해결하기 어렵고, ML이나 딥러닝이 강점을 발휘할 수 있습니다.

---

## 좋은 ML 프로젝트 시작 순서

강의에서 제안하는 좋은 시작 방식은 다음과 같습니다.

```text
1. 문제를 정의한다
2. 간단한 휴리스틱을 만든다
3. 선택한 평가 지표로 휴리스틱 성능을 측정한다
4. 휴리스틱 성능을 baseline으로 둔다
5. 간단하고 해석 가능한 ML 모델을 만든다
6. ML 모델과 휴리스틱 baseline을 비교한다
7. 성능 향상이 충분할 때만 더 복잡한 모델로 확장한다
```

처음부터 복잡한 딥러닝 모델로 가는 것은 좋은 방식이 아닙니다. 먼저 평균값, 최빈값, 간단한 비즈니스 룰 같은 기준선을 만들고, 이후 선형회귀, 로지스틱 회귀, 간단한 의사결정나무 같은 해석 가능한 모델로 시작하는 것이 좋습니다.

---

## Baseline이 중요한 이유

Baseline은 모델 성능을 판단하기 위한 비교 기준입니다.

ML 모델을 만들었는데 성능이 좋아 보이더라도, 단순 규칙보다 거의 나아지지 않았다면 도입할 이유가 약합니다. 예를 들어 복잡한 모델의 정확도가 82%이고, 단순 휴리스틱의 정확도가 80%라면 2% 향상이 비용과 복잡도를 정당화하는지 따져봐야 합니다.

> 좋은 ML 프로젝트는 “모델을 만들었다”가 아니라, “간단한 기준선보다 명확히 더 나은 가치를 만들었다”를 증명해야 합니다.

---

## ML 도입 전 체크 질문

### 1. 휴리스틱만으로 충분한가?

간단한 규칙만으로도 사용자가 만족할 만한 성능이 나온다면 ML을 도입하지 않아도 됩니다.

### 2. ML이 baseline보다 충분히 나은가?

ML 모델이 휴리스틱보다 좋아야 합니다. 단순히 조금 좋아지는 수준이 아니라, 복잡도와 비용을 감당할 만큼 좋아야 합니다.

### 3. 성능 향상이 비즈니스 가치로 이어지는가?

정확도나 예측 성능이 올라도, 그것이 사용자 경험 개선, 비용 절감, 품질 향상, 의사결정 개선으로 이어지지 않으면 의미가 약합니다.

### 4. 틀렸을 때의 비용이 감당 가능한가?

의료, 채용, 금융, 안전 관련 문제처럼 잘못된 예측의 피해가 큰 경우에는 ML 자동화를 특히 신중하게 판단해야 합니다.

### 5. 운영과 유지보수를 감당할 수 있는가?

ML은 배포 후에도 끝이 아닙니다. 데이터 변화, 성능 저하, 재학습, 모니터링을 계속 관리해야 합니다.

---

## 실무적으로 중요한 판단 기준

ML 도입 여부는 기술 욕심이 아니라 **성능 향상 대비 비용과 리스크의 문제**입니다.

```text
문제 정의
→ 사용자에게 필요한 가치 확인

휴리스틱 baseline
→ 단순한 방법으로 어디까지 가능한지 확인

간단한 ML 모델
→ ML이 baseline을 얼마나 넘는지 확인

복잡한 모델
→ 추가 복잡도가 그만한 가치가 있는지 판단
```

즉, “ML을 쓸 수 있는가?”보다 “ML을 써야 하는가?”가 먼저입니다.

---

## 헷갈리기 쉬운 포인트

### 휴리스틱은 허접한 방법이 아니다

휴리스틱은 대충 만든 임시방편이 아니라, ML 프로젝트에서 반드시 필요한 기준선입니다. 기준선이 있어야 ML의 성능 향상이 의미 있는지 판단할 수 있습니다.

### 좋은 모델은 복잡한 모델이 아니다

좋은 모델은 문제를 충분히 잘 해결하면서도 운영 가능한 모델입니다. 간단한 모델이 충분히 좋은 성능을 낸다면 그것이 더 좋은 선택일 수 있습니다.

### ML 성능 향상은 비용과 함께 봐야 한다

정확도가 조금 올랐다고 항상 좋은 것은 아닙니다. 그 성능 향상이 데이터 준비, 개발, 컴퓨팅, 운영, 모니터링 비용을 정당화해야 합니다.

### 자동화와 보조는 다르다

ML이 사람을 완전히 대체할 수도 있지만, 때로는 사람의 판단을 보조하는 방식이 더 안전하고 가치 있을 수 있습니다.

---

## 프로젝트 / 면접에 써먹을 수 있는 문장

ML 프로젝트를 시작할 때는 먼저 휴리스틱 기반 baseline을 구축하고, 동일한 평가 지표로 ML 모델과 비교해야 합니다. ML 모델이 baseline 대비 충분한 성능 향상과 비즈니스 가치를 제공할 때만 복잡한 모델 개발과 운영 비용이 정당화됩니다.

---

## 최종 정리

이 강의의 핵심은 ML을 무조건 도입하는 것이 아니라, **사용자 가치와 비용 대비 효과를 기준으로 ML 도입 여부를 판단하는 것**입니다.

ML은 자동화, 예측, 개인화를 통해 가치를 만들 수 있지만, 동시에 환경 변화, 윤리, 책임 소재, 높은 오답 비용 같은 리스크를 가집니다.

따라서 좋은 ML 프로젝트는 먼저 휴리스틱 baseline을 만들고, ML이 그 기준선을 충분히 넘어설 때만 더 복잡한 모델로 나아가야 합니다.
